REST API Concept Notes

A complete guide to REST architecture, HTTP verbs, status codes, database operations, and auth patterns.  ← Back to Lab  ·  Open Practice Page

1. What is REST?

REST (Representational State Transfer) is an architectural style for building distributed systems over HTTP. It was defined by Roy Fielding in his 2000 PhD dissertation. REST is not a protocol or standard — it's a set of constraints that, when followed, produce scalable, stateless, cacheable web services.

A service that fully follows REST constraints is called RESTful. The key idea: everything is a resource, and clients interact with resources using a uniform interface (HTTP methods + URLs).

The 6 REST Constraints

ConstraintMeaning
Client-ServerUI and backend are decoupled. They communicate only via the API. Each can evolve independently.
StatelessEvery request must contain all information needed to process it. The server stores no client session state between requests.
CacheableResponses must label themselves as cacheable or non-cacheable. Improves scalability and performance.
Uniform InterfaceA single, consistent way to interact with all resources (standardised URLs + HTTP methods + JSON).
Layered SystemClients don't need to know if they're talking directly to the server or through a proxy, CDN, or load balancer.
Code on Demand (optional)Server can return executable code (e.g. JavaScript). Rarely used.

2. HTTP Methods (Verbs)

REST uses standard HTTP verbs to express what action to perform on a resource.

MethodActionSafe?Idempotent?Body?
GET Read a resource (or collection) ✅ Yes✅ Yes❌ No
POST Create a new resource ❌ No❌ No✅ Yes
PUT Replace a resource entirely ❌ No✅ Yes✅ Yes
PATCH Partially update a resource ❌ No✅ Yes✅ Yes
DELETE Remove a resource ❌ No✅ Yes❌ No
Safe = does not modify data.    Idempotent = calling it N times has the same effect as calling it once.

PUT vs PATCH — key difference

// PUT — send the entire object (missing fields reset to defaults)
PUT /api/products/1
{ "name": "Laptop Stand", "category": "electronics", "price": 44.99, "stock": 50 }

// PATCH — send only the fields you want to change
PATCH /api/products/1
{ "price": 44.99 }   // only price changes; everything else stays the same

3. HTTP Status Codes

Status codes tell the client what happened to their request.

CodeMeaningWhen to use
200 OKSuccessGET, PUT, PATCH succeeded
201 CreatedNew resource createdPOST succeeded; include Location header
204 No ContentSuccess, no bodyDELETE succeeded
400 Bad RequestClient sent invalid dataValidation failed
401 UnauthorizedNot authenticatedMissing or invalid token
403 ForbiddenNot authorisedToken valid but role insufficient
404 Not FoundResource doesn't existID not in DB
409 ConflictState conflictDuplicate username, unique constraint
500 Internal Server ErrorBug on the serverUnhandled exceptions

4. URL Design Principles

Use nouns, not verbs

// ❌ Bad — verbs in URL
GET /getProducts
POST /createProduct
DELETE /deleteProduct/5

// ✅ Good — nouns + HTTP verb carries the action
GET    /api/products
POST   /api/products
DELETE /api/products/5

Nested resources for relationships

// Reviews belong to a product
GET    /api/products/1/reviews      // list reviews for product 1
POST   /api/products/1/reviews      // add a review to product 1
DELETE /api/reviews/42              // delete review #42 (flat, still ok)

Query strings for filtering, searching, sorting

GET /api/products?category=electronics&min_price=20&sort=price&order=asc&page=2&limit=10
Keep URLs lowercase, use hyphens not underscores, be consistent with plurals (/products not /product).

6. Authentication with JWT

This lab uses JSON Web Tokens (JWT). The flow is:

  1. Client sends POST /api/auth/register or POST /api/auth/login with credentials.
  2. Server verifies credentials and returns a signed JWT.
  3. Client stores the token and sends it in every subsequent request as: Authorization: Bearer <token>
  4. Server verifies the token signature on protected routes. If invalid → 401.

JWT structure

// Three base64-encoded parts separated by dots
eyJhbGciOiJIUzI1NiJ9    ← Header (algorithm)
.eyJpZCI6MSwidXNlcm5hbWUiOiJhbGljZSIsInJvbGUiOiJhZG1pbiJ9  ← Payload
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c               ← Signature

7. Database Operations in This Lab

The server uses SQLite (file-based SQL database) via the sqlite3 Node.js driver. Four tables are created automatically on first start.

Schema

users         id, username, password_hash, role, created_at
categories    id, name, description, created_at
products      id, name, category, price, stock, description, created_at, updated_at
reviews       id, product_id, user_id, rating, comment, created_at

Key SQL patterns used

-- CRUD: basic operations
SELECT * FROM products WHERE id = ?
INSERT INTO products (name, category, price) VALUES (?,?,?)
UPDATE products SET price=?, updated_at=CURRENT_TIMESTAMP WHERE id=?
DELETE FROM products WHERE id = ?

-- Filtering + pagination
SELECT * FROM products
  WHERE category = ? AND price >= ? AND (name LIKE ? OR description LIKE ?)
  ORDER BY price ASC
  LIMIT 10 OFFSET 20;    ← page 3 of 10 results

-- JOIN (products + reviews)
SELECT p.name, ROUND(AVG(r.rating),1) AS avg_rating, COUNT(r.id) AS reviews
FROM products p
LEFT JOIN reviews r ON r.product_id = p.id
GROUP BY p.id
ORDER BY avg_rating DESC;

-- Aggregation (GROUP BY + aggregate functions)
SELECT category, COUNT(*) AS count, MIN(price), MAX(price), AVG(price), SUM(stock)
FROM products
GROUP BY category;

-- Transaction (bulk insert — all or nothing)
BEGIN TRANSACTION;
  INSERT INTO products ...;
  INSERT INTO products ...;
COMMIT;

Stock management pattern

-- Atomic increment (safe for concurrent requests)
UPDATE products SET stock = stock + ?, updated_at = CURRENT_TIMESTAMP WHERE id = ?

-- Check-then-decrement (prevents negative stock)
-- 1. Read current stock
SELECT stock FROM products WHERE id = ?
-- 2. If stock >= qty → proceed
UPDATE products SET stock = stock - ? WHERE id = ?

9. Quick Start

# 1. Install dependencies
cd labs/rest-api
npm install

# 2. Start the server
npm start
# → REST API Lab running → http://localhost:4101

# 3. Register an admin user
curl -X POST http://localhost:4101/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"secret123","role":"admin"}'

# 4. Login and capture the token
curl -X POST http://localhost:4101/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"secret123"}'

# 5. Seed sample data (using the token from step 4)
curl -X POST http://localhost:4101/api/seed \
  -H "Authorization: Bearer <token>"

# 6. List products with filters
curl "http://localhost:4101/api/products?category=electronics&sort=price&order=asc"
Open http://localhost:4101 in your browser for the fully interactive UI — no curl needed.
